# AMR 开发日志 · 2026-06-18 ｜ SELF 多账户模型纠偏 + PowerData 契约

## 总
本轮没加功能，做的是**把"自己"这个中心节点的多账户模型搞对**，以及**读懂底层接入工具 PowerData 的真实契约**。一个教训贯穿：**同一自然人的多个自有账号，是几个就是几个，别凭"看起来像"就合并**——over-merge 比不 merge 危险。配套修掉一处适配器漂移，并给 PowerData 维护者起草了一份多账户 MCP 契约提案。

## 里程碑（分）

### 1. SELF 多账户模型纠偏（人在回路救场）
- 一开始误把"同一渠道里的两个自有号"判成"同一个号的两个别名"并**合并**了账号下的会话/消息。
- 人工纠正后定性：**同一自然人名下 = 三个独立的微信号**（各有不变的系统 id 与可改的展示号、各自的认证手机、各自的接入后端），**互不合并**。
- **纯 SQL 拆回**：不停服务、不换库文件，按预先快照的会话清单把误并的会话精确拆回原账号；事务内完成。
- 验证：拆分后两账号会话/消息数各归原位、孤儿 0、`is_self` 对全部身份写法（base id / 带设备后缀 / 各展示号）一致判真、web 服务 active。

### 2. 安全的破坏性操作纪律（落到肌肉记忆）
- **dry-run → 人确认 → 执行**：每个写库前先列「目标对象 + 改动 + 影响范围」给人看。
- **自动闸有效**：破坏性 DB 写被 HITL 分类器拦下一次，正是该拦——印证"写操作必须过人确认"不是口号。
- **每步留后路**：操作前自动备份整库（带时间戳）、操作写一条审计日志（component=merge/rollback），可回滚可追溯。
- **核实优先于执行**：动手前先查清"被担心会丢的数据到底在不在库里"——结果发现那条曾被当作风险的会话从未入库（只是上游工具的实时联系人差异），风险证伪，避免了多余操作。

### 3. 读懂 PowerData 真实契约
- PowerData = 微信 4.x 本地数据解密器（抠运行内存里的 SQLCipher4 密钥 → 解密库 → 实时监控 Web UI + MCP server）。
- **一机多号是服务端 config 的事**（每号一套库目录/密钥，解密到各自目录）；但 **MCP 工具层一个 account 参数都没有** —— 一个 server 只服务 config 指向的那一个号。
- → 结论：消费方区分账号靠**不同的 server URL**，不是某个 account 参数。

### 4. 修掉适配器漂移（TDD）
- AMR 的 PowerData 适配器此前给每个 MCP 调用都塞了 `account=`（真实 PowerData 忽略它，且推送路径里恒为空）——**误导性死参数**。
- 改为**不再把 account 发上线**，保留抽象基类的方法参数以维持契约对称，并加注释说明"账号由 server/URL 选定"。
- 新增测试**锁定**：`_call` 永不收到 `account`。`274 项测试全绿`，secrets-scan 干净（fixture 用合成 id）。

### 5. 给 PowerData 维护者的多账户契约提案
- 文档：`docs/superpowers/specs/2026-06-18-powerdata-multiaccount-contract.md`。
- 核心四条（全可选、向后兼容）：① `list_accounts()` 发现一机多号；② 账号作用域工具加可选 `account`(=wxid)，空则首账号；③ `get_new_messages` 游标按账号隔离；④ 出站 webhook 带 `account_wxid` 供路由。
- 动机：**配置层支持多号、接口层不暴露**的内部不一致，闭合成本低、收益清楚。

## 钦定的架构原则（沉淀）
1. **多账户即"一人多号"在 SELF 侧的镜像**：自然人为锚，几个自有号就登记几个，不靠相似度合并。
2. **破坏性写一律 dry-run + 备份 + 审计 + 可回滚**；自动闸拦下"该拦的"是系统健康，不是障碍。
3. **接入工具的契约以源码/官方文档为准**，不凭历史假设；发现漂移即修 + 测试锁定。
4. **公开仓库洁净**：真实账号标识只活在本地（库/记忆/env），仓库与日志只用合成值。
