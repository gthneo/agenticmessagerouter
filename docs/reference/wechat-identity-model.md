# 微信身份标识技术综述 — 六个标识 + 与 AMR 账号模型的映射

## 总述

微信对「一个账号」其实挂了**六个不同的标识**，分三层。把它们混为一谈，是身份归一一切 bug 的根源（AMR 早期把昵称当 id 存就是一例）：

- **内部层（不可变·路由用·UI 不露）**：`wxid`（WeChat 内部账号 ID·字符串账号指纹）、`uin`（User Identification Number，用户识别号·数字账号号；腾讯账号体系从 QQ 沿用，全称为业界通用展开、非官方文档）。
- **显示层（可变·人设·有可见性差异）**：`昵称 nickname`、`微信号 WeChat-ID`、`备注名 remark`。
- **绑定层（可换·身份证明）**：`手机号`（外加邮箱/QQ 等，历史）。

**最关键的一条认知**：六个里**只有 `wxid` / `uin` 对「账号」永久且唯一**，其余四个都会变、会撞、或仅本地可见。所以**账号的稳定锚必须是 wxid/uin，绝不能是昵称/微信号/备注名**。而「账号 → 人」是多对一（一个人多个账号 = 多个 wxid，彼此无算法关联），所以**人层归一必须人显式声明，不能算法推断**（详见 §6）。

| 标识 | 谁设 | 可变性 | 可见性 | 对账号唯一/稳定 | 主要用途 |
|---|---|---|---|---|---|
| **wxid** | 系统(注册时) | **永不变** | **看情况**：对方没设自定义微信号时，微信号字段**直接露出 wxid**（UI 可见）；设了则隐藏。本地 DB 恒有 | ✅ 唯一+永久 | 协议路由主键、跨设备/会话锚 |
| **uin** | 系统(注册时) | **永不变** | 不可见(内部) | ✅ 唯一+永久 | 数字账号号，与 wxid 同账号两种表示 |
| **昵称** | 账号主人 | 随便改 | **他人可见** | ❌ 可变可撞 | 展示名 |
| **微信号** | 账号主人 | 可改(有频次限制) | **他人可搜可见** | ⚠️ 可变；老号曾=wxid | 可搜索的对外 ID；**可内嵌手机号** |
| **备注名** | **你**(对某联系人) | 随时改 | **仅你本地可见** | ❌ 你的私有标注 | 你给对方起的名 |
| **手机号** | 账号主人 | 可换绑 | 取决于隐私设置 | ⚠️ 可换、可被多账号复用 | 注册/登录/找回身份证明 |

> 截图三字段对照（合成示例）：`备注名:老李` / `昵称:李工` / `微信号:kerwin_test`。三者是**三个独立字段**，不是一个名字的三种叫法。

## 1. 显示层三字段 —— 昵称 / 微信号 / 备注名

最容易被系统糊成「一个 name」的三个，其实语义、可见性、归属都不同：

| | 昵称 nickName | 微信号 WeChat-ID | 备注名 remark |
|---|---|---|---|
| 谁设的 | **对方**自己设 | **对方**自己设 | **你**给对方设 |
| 存在哪 | 对方账号 + 你本地缓存 | 对方账号 | **只在你本地** |
| 他人可见 | 是 | 是(可搜索) | **否(仅你)** |
| 变更 | 随时，无限次 | 有频次/规则限制 | 随时 |
| 会撞吗 | 极易(同名一片) | 唯一(全局不重) | 你自己说了算 |
| 风控/推断价值 | 低 | **中(见下)** | 无(私有) |

**微信号能反推手机号（隐私/推断向量·重要）**：微信号是用户自取的字符串，很多人**直接拿手机号当微信号**或嵌进去 —— 形如 `li13000000000`（前缀 + 11 位手机号，合成示例）。于是**从微信号字符串能直接读出/猜出手机号**。昵称里也常含手机号尾号/全号。归一与隐私处理时要把「微信号/昵称含 11 位数字串」当**疑似手机号**识别并谨慎（既是归一线索，也是 PII 泄漏面）。

**备注名是你的私有真相**：它是你对这个人的认知锚（"老李·液冷供应商"），**比对方的昵称/微信号更稳、更贴你的关系语义**。AMR 把它解析掉（PowerData 路径只留 id token，`powerdata.py:113-123`）是丢了最有价值的关系标注 —— 见 §7。

## 2. 内部层 —— wxid 与 uin（不可变锚）

### 2.1 wxid（字符串账号指纹）
- **现代号（约 2014 年后注册）**：格式 `wxid_` + 约 14~15 位小写字母数字，例 `wxid_test_abc123def456`。
- **UI 可见性 = 取决于对方有没有设自定义微信号**（重要，别当成"永不可见"）：
  - **没设自定义微信号** → 资料页的「微信号」字段**直接回落显示 wxid_ 本身** → 你在 UI 上**就能看到对方 wxid**（合成例：一个没设微信号的联系人，资料页「微信号：wxid_test_abc123」）。
  - **设了自定义微信号** → UI 显示那个自定义号、隐藏 wxid。
  - 所以「联系人的微信号长得像 `wxid_xxx`」= 强信号：TA 没设微信号，且你**直接拿到了 TA 的 wxid**。
- **即便被隐藏，也在本地 DB 里**：微信协议内部**用 wxid 做路由主键**，你本地 DB 存着你自己 + 你所有联系人的 wxid（联系人表、每条消息的发件人字段）。拿到本地库 + 解密密钥（后端 PowerData / agent_server 干的事）就能读出。**边界：只能读到你数据范围内的（你自己+你联系人），陌生人的取不到。**
- **老号（早期注册）的陷阱**：约 2014 年前注册、且早早设过自定义微信号的账号，其**内部 wxid 往往就是当年第一次设的那个微信号字符串本身**（无 `wxid_` 前缀）。之后你把对外微信号、昵称都改了，**那个内部 wxid 串不变**。结果：老号的 wxid 长得像个微信号/昵称 → 诱导系统把「微信号/昵称/wxid」混为一谈。
- **永久性**：改昵称、改微信号、换绑手机号、换设备、客户端 3.x→4.x 升级 —— **wxid 全程不变，且退役后基本不重新分配给别人**。它是体系里唯一可靠的稳定锚。

### 2.2 uin（User Identification Number，用户识别号·数字账号号）
- 除了字符串 wxid，每个账号还有一个**纯数字的内部账号号（uin = User Identification Number，用户识别号；腾讯从 QQ 账号体系沿用，QQ 号本质就是 uin）**，协议某些层用它。**uin 与 wxid 指向同一账号、是同一身份的两种表示**，同样注册即定、终身不变。
- uin 在本地 DB 里是**加密存储**的，不在任何 REST 表面暴露，需解密账号自身的 contact/session 库才能拿到。
- **QQ 迁移号的推断向量**：早期（约 2011~2014）大量微信账号由 QQ 迁移/绑定而来，用户常把**微信号设成「姓拼音首字母 + QQ 号」**（合成例：`w1000000000` = 王 + QQ 号 `1000000000`）。**QQ 号本身就是 QQ 时代的 uin** → 这类微信号**直接暴露了该号的 QQ-uin 血脉**（能否等同于该微信账号当前的内部 uin 需解密核实，但为强线索）。同 §1 的「微信号可内嵌手机号」，是归一线索 + PII 泄漏面。

### 2.3 没有官方生成规范
**腾讯不公开 wxid/uin 的生成算法**，它们是内部专有标识，不在任何官方开发者文档里。本节的「格式/前缀/老号=微信号」均为**大量样本逆向与观测**所得，**不是官方口径**；那串后缀看不出递增规律、无人类语义，像服务端生成的 token，但确切算法**不可知**。

## 3. 手机号 —— 绑定层与「前后手机号如何关联」

- **手机号是绑定层、不是身份本身**：账号靠 wxid/uin 立身，手机号只是**注册/登录/找回的凭证**，**可换绑**。
- **前后手机号怎么关联**：同一账号**换绑手机号**时，**wxid/uin 不变** —— 所以「旧手机号」和「新手机号」之所以是「同一个人」，**靠的是它们先后绑定到同一个 wxid/uin**，不是手机号之间有什么直接关系。要追一个人的手机号变迁史，**锚在 wxid/uin、把手机号当该账号的时间序属性**，而不是反过来用手机号当主键（手机号会被注销、回收、再分配给陌生人）。
- **一号多绑 / 一人多号**：一个手机号在不同时间可能绑过不同账号；一个人也常用多个手机号注册多个微信账号。**手机号 ↔ 账号是多对多的时间序关系**，不是稳定一一对应。
- **归一价值与风险**：手机号是跨渠道归一的强线索（微信账号 ↔ 通讯录电话 ↔ 企微 external_userid 往往同号），但因可换绑/可复用，**只能作人显式归一的「证据」，不能作自动合并的「判据」**。

## 4. 设备 / 实例后缀（`_2325` / `_e96b`）

多开 / 多设备 / 自动化层（PowerData、agent_server）会在 wxid 后**追加一段后缀**来区分会话/实例：`wxid_test_abc_2325`、`wxid_test_abc_e96b`。

- **后缀不是 wxid 的一部分**，是接入层加的实例号。
- **剥掉后缀按同 base wxid 合并，是唯一安全的「自动」合并**（同一账号的多开/多端，必是同一个人）。AMR 的 `_canon_identifier`（`src/jl/db.py:234`，正则 `_WXID_SUFFIX` 见 `db.py:231`）只做这一件事：把 `wxid_<core>_<1~12 位后缀>` 收敛到 `wxid_<core>`；`onboard._base_wxid`（`onboard.py:175-183`）在预检里用更强的前缀剥法兜多下划线 base。
- **注意**：自定义微信号形态的老号 wxid（无 `wxid_` 前缀）**不被正则触及**（保守），故老号的实例后缀需另行人审。

## 5. 映射到 AMR 数据模型（现状 + 缺口）

AMR 库结构（`src/jl/schema.sql`）：`persons`(自然人) ← `channels`(可达端点) / `conversations`(会话) ← `messages`；`accounts`(你自己的登录) 与 `self_identities`(你自己的跨渠道身份) 是另两轴。

| 微信标识 | AMR 现有落点 | 状态 |
|---|---|---|
| **wxid** | `channels.identifier`(kind=wechat) / `conversations.chat_id` / `messages.sender_id` / `self_identities.identifier`；经 `_canon_identifier` 收敛 | ✅ 一等公民 |
| **昵称** | 塌进唯一的 `channels.label`/`conversations.name`/`messages.sender` 显示槽 | ⚠️ **与备注名塌缩**，无独立列 |
| **备注名** | 同上一个显示槽；PowerData 路径**直接解析掉**(`powerdata.py:113-123`) | ❌ **被丢/被塌缩** |
| **微信号** | **无专列**；只能塞 `channels.meta`(JSON)，或更糟——被当成 `identifier`(PowerData 首 token 若是自定义号)→ 不收敛、配不上 fullwechat 的真 wxid → **归一失败** | ❌ **无位+有害混淆** |
| **uin** | 无任何列、代码零引用 | ❌ **完全未建模** |
| **手机号** | 独立 `channels.kind='phone'` 行(`canon_phone` 规范化)，靠同 `person_id` 与微信身份关联 | ⚠️ 与微信身份**未直连**，仅经 person 旁挂 |

**一句话**：AMR 今天的微信身份 = **{ 一个 wxid + 一个不分化的显示名 }**，三个名字概念塌成一个串、uin 没影、备注被丢。这是 §7 要补的。

## 6. 归一原则（王总 2026-07-01 钦定）

1. **两层模型**：**账号层**每个 wxid/登录各一行、**保留来源**（哪个后端 / DB 世代 / PowerData 实例）、不塌缩；**人层(person/SELF)** 多账号归一。
2. **人层人显式合并**：一个人的多个微信账号（多个无关 wxid）之间**没有可推断的桥**，**必须人显式声明「这几个号都是我/都是 TA」**（AMR 走 `self_identities` + `mark_person_self` + `suggest_merges` 的 HITL 候选，`db.py:909-1005`）。
3. **绝不靠昵称/不同 wxid 自动合**：昵称会变会撞、不同 wxid 无关联；自动合 = 污染身份图。唯一自动合 = **同 base wxid 剥后缀**（§4）。
4. **老号陷阱**：wxid 长得像微信号时，别把它当微信号/昵称处理；它仍是不可变锚。
5. **PowerData 3.x/4.x 双读 → 跨版本去重**：PowerData 同时能读 3.x 与 4.x 库，给的是 4.x 数据但运行时把同机残留的 3.x 数据也取了一遍 → **同一对话/消息可能跨版本双份入库（id 不同）**。person 层归一时要做跨版本去重（与 `msg_id` 全局唯一、读哑失败同属一个「响亮 > 静默」治理家族）。

## 7. 缺口与建议（→ 跟进任务）

当前 AMR 把 `昵称 / 微信号 / 备注名` 塌成一个显示槽、`uin` 不建模、PowerData 丢备注。建议（独立任务实现，TDD）：

- **拆分显示字段**：`昵称` / `备注名` 各自独立（备注名是你的私有关系真相，最该留）；不再让后端随便一个串覆盖单 `name`。
- **`微信号` 给专列**（而非塞进 `identifier` 造成 merge 失败）；识别「微信号/昵称含 11 位数字」= 疑似手机号（归一线索 + PII 提示）。
- **`uin` 建模**：作为与 wxid 并列的账号不可变锚（之一），尤其老号「wxid=微信号」时 uin 是更干净的锚。
- **来源留痕**：账号/会话记下 DB 世代与后端实例，支撑跨版本去重。
