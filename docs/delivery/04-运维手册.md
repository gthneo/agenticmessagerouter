# AMR 运维手册（受众：数字运维工程师 Agent + 现场 Build 工程师）

> 适用版本 **v0.11.0** ｜ 更新日 2026-06-28
> 写法：面向 **Agent 可直接执行**——命令块 + 排障表 + 凭据路径。**凭据不入库**：
> SSH/token 等真值见运维方本地笔记（不在公开仓库）。部署主机 IP/端口为内网默认。

## 部署 / 升级
```sh
# 从开发机推代码 + 刷 systemd --user 服务（幂等）
deploy/deploy.sh <user>@<deploy-host>
# 等价手动：rsync src/jl pyproject.toml deploy → ~/amr/，再 bash ~/amr/deploy/install.sh
```
- 零运行时依赖（stdlib），系统 python3.10 跑，`PYTHONPATH=src`。
- DB：`~/.config/jl/jl.db`（真相源，**不入库**）。Schema 变更经 `db.init_db` 幂等 ALTER
  (`_ADDED_COLUMNS`) + 新表 `CREATE TABLE IF NOT EXISTS` 自动迁移，无需手动 migrate。

## 服务（systemd --user，enable-linger 重启存活）
```sh
systemctl --user status|restart amr-web amr-poll
journalctl --user -u amr-web -n 100 --no-pager
```
- `amr-web`：只读+操作 web 收件箱，:8088（token 在 `~/.config/jl/web_token`）。
- `amr-poll`：5 分钟拉新 + 自动拟稿 sweep + 主动 opener sweep（均 LLM-optional，永不自动发）。
- 环境：`~/.config/jl/amr.env`（`JL_WEB_TOKEN` / `AMR_LLM_PROVIDER=claude_code` /
  `AMR_CLAUDE_BIN`=绝对路径）。install.sh 首次生成 web_token + 写 amr.env。

## 常用运维命令（jl CLI）
```sh
jl                      # 关系账户健康度 sweep（染色红榜）
jl 主动 [名]            # 主动联络队列 / 给单人拟开场
jl 关注 [--off] <名>    # 标记/取消关注（进主动队列）
jl 连 <名> <chat_id>    # 把人连到正确的实时微信会话（修缺渠道/换正确端点）
jl 关注 [--off] <名>    # 标记/取消关注（进主动队列）
jl 电话归一             # 折叠同号 +86/格式变体重复会话；号码 canon 去重
jl 解绑 <conv>          # 拆开错误归并（把某会话从当前 person 解开，归一 v2）
jl --tokens            # LLM token 用量统计
```
端点维护（python，按需）：`db.dedup_channels(c)`（折叠渠道表同号端点）、
`db.dedup_phone_conversations(c)`（折叠会话表）、`db.set_endpoint_pin(c, pid, kind, ident, True)`（钦点首选发送端点）。

## 自我身份 + 归一（v0.8，全人在回路）
```sh
# 自我身份：首启自动从 accounts.self_id 种子；建议+勾选见 ⚙ 设置 / API
#   db.seed_self_from_accounts(c)        # 把 accounts.self_id 纳入登记（persona=自我）
#   db.suggest_self_identities(c)        # 自动建议候选（不自动入，人勾选）
#   db.add_self_identity(c, kind, ident, persona='自我', label='')
#   db.apply_self_directions(c)          # 把 sender∈self 的消息标 direction=out（"我"），ingest 后调，幂等
# 归一（reunify）：
#   db.reunify(c)                        # 全扫：强信号自动连 + 弱信号留候选
#   db.reunify(c, reset=True)            # 复位：只清【自动连】（不动人工确认），重跑——需二次确认
```
- 自我会话（`chat_id ∈ self_identities`）被 `proactive_sweep`/openers/`persons_overview`
  **自动跳过**——不给自己拟开场、不进「该联系谁」/联系人。
- Web 设置 API（人在回路，留痕 events）：`GET/POST /api/self`、`POST /api/self/remove`、
  `POST /api/connect`、`POST /api/unlink`、`POST /api/reunify`（`?reset=1` 需二确）。
- **registry seeded from account self_id**：渠道命令的本人身份从 `accounts.self_id` 自动播种，
  不手填真值（凭据/真值不入库）。

## 排障表（dogfood 实战）
| 症状 | 原因 | 处理 |
|---|---|---|
| 发送「No action selected」/「TA 不在微信近期会话」 | 目标 chat_id 不在 fullwechat 实时可选列表（裸 wxid/已冷） | 在微信里打开与 TA 的对话使其进近期列表，或 `jl 连 <名> <实时id>` 重连正确端点 |
| 时间线出现 `<msg>cdnthumb…` 原始 XML | 旧数据媒体未清洗 | 用 `fullwechat.clean_content` 按 raw 重算 content（见 git 历史的清洗脚本）；新数据 ingest 已自动清 |
| 一人多条重复电话/渠道 | +86/格式变体未归一 | `jl 电话归一` + `db.dedup_channels` |
| 两个人被错并成一个 | 归一 v2 误连/弱信号被接受 | `jl 解绑 <conv>` 拆开；或 `db.reunify(c, reset=True)`（二确，只清自动连） |
| 本人出现在「该联系谁」/给自己拟了开场 | 自我身份未声明 | ⚙ 设置勾选认领，或 `db.add_self_identity`；`db.apply_self_directions(c)` 重标方向 |
| AI 话术/诊断「未配置 LLM」 | claude CLI 未登录 / amr.env 缺 AMR_CLAUDE_BIN | 检查 `~/.config/jl/amr.env`；`claude` 须以运维账号登录（Max plan）；人手操作不受影响 |
| 发送被拒（auto 模式拦截） | HITL 闸：禁止 Agent 代发 | 正常——由人在 web UI 点「确认发送」 |
| git push 卡住 | 受限网络 HTTPS 上传慢，勿用短超时掐断 | `GIT_HTTP_LOW_SPEED_TIME=20 git push`，给 60s+，别自掐 |

## 凭据轮换
- web token：删 `~/.config/jl/web_token` 后重跑 install.sh 重生成。
- fullwechat / claude / lark 凭据：见运维方本地笔记，按各渠道流程轮换；**不写入仓库**。

- reboot 存活：`loginctl enable-linger <user>` 已开（install.sh 设）。

## v0.10.0 运维 delta（今日简报 + 可换皮肤 + 手机端）

### 升级到 v0.10.0

```sh
# 从开发机推代码（增量，不覆盖 DB）
rsync -av --exclude='*.db' src/jl/ <user>@<host>:~/amr/jl/

# 重启两个服务
ssh <user>@<host> "systemctl --user restart amr-web amr-poll"

# 验收
ssh <user>@<host> "systemctl --user status amr-web amr-poll"
```

> **PYTHONPATH 陷阱**：部署目标上运行包是 `~/amr/jl`（由 `CWD=~/amr python3 -m jl.cli` 导入），
> **不是** `~/amr/src`。rsync 只推 `src/jl/` 内容到 `~/amr/jl/`，不要推整个 `src/`。

> **DB 铁律**：`rsync` 命令必须带 `--exclude='*.db'`，绝不覆盖 `~/.config/jl/jl.db`（真相源）。

### 服务说明（v0.10.0）

| 服务 | 作用 | 端口/路径 |
|---|---|---|
| `amr-web` | Web 收件箱 + 今日简报 API（只读+操作） | `:8088`，token 见 `~/.config/jl/web_token` |
| `amr-poll` | 5 min 增量拉新 + 自动拟稿 sweep | 无端口，日志 `journalctl --user -u amr-poll` |

两个服务均以 systemd `--user` 模式运行，`loginctl enable-linger` 保证 reboot 后存活。

### 今日简报接口健康检查

```sh
# 验证 /api/digest 返回 200 JSON（替换 <token> 为实际 token）
curl -s "http://localhost:8088/api/digest?token=<token>" | python3 -m json.tool | head -30
```

正常响应为包含 `sales` / `relations` / `marketing` / `matters` / `meta` 五个键的 JSON 对象。
若某维度后端尚未实现，对应键值为 `{"status": "pending"}` 占位，不报错。

### web token 轮换

```sh
# 1. 删旧 token 文件
ssh <user>@<host> "rm ~/.config/jl/web_token"

# 2. 重跑 install.sh 自动重生成 + 写入 amr.env
ssh <user>@<host> "bash ~/amr/deploy/install.sh"

# 3. 重启 amr-web 使新 token 生效
ssh <user>@<host> "systemctl --user restart amr-web"

# 4. 把新 token 交付客户（不入库、不明文写在任何公开记录）
ssh <user>@<host> "cat ~/.config/jl/web_token"
```

### 可换皮肤说明（运维视角）

皮肤切换完全在客户端（浏览器 `localStorage['amr_skin']`），**无需后端操作**。
- 合法值：`digest`（今日简报，默认）、`inbox`（旧三栏收件箱）。
- 若需给某用户强制默认皮肤：在 `amr-web` 入口页注入 `<script>localStorage['amr_skin']='inbox'</script>`（按需，非默认操作）。
- 手机端（窄屏）忽略 `amr_skin`，自动呈现单列卡片流 + 底部 tab。

### v0.10.0 排障补充

| 症状 | 原因 | 处理 |
|---|---|---|
| `/api/digest` 404 | amr-web 服务版本老于 v0.10.0 | 确认 rsync 已推最新 `src/jl/`，重启 `amr-web` |
| 简报某卡片显示 `{"status": "pending"}` | 该维度后端未实现 | 预期行为，向客户说明"待后端接入，非报错" |
| 浏览器打开仍是旧三栏 | `localStorage['amr_skin']` 残留旧值 | 浏览器开发者工具 → Application → localStorage → 删 `amr_skin` 键，刷新 |
| 手机底部 tab 未出现 | 屏幕宽度超过响应式断点（如平板横屏） | 竖屏使用；或缩小浏览器窗口宽度至 < 768px |

## v0.11.0 运维 delta（运维 API + 版本规范 + 接账号身份预检 + 自动回 killswitch + 发送排障）

### 1. PII-free 运维 API（给外部数字运维工程师 Agent）

> 完整对接手册：`docs/runbooks/amr-ops-api-for-ops-agent.md`。

AMR 新增**公开、无令牌、零 PII** 的运维健康端点，专供**外部监控 Agent** 7×24 轮询告警：

```sh
curl -s "http://<AMR_HOST>:8088/api/health" | python3 -m json.tool
```

返回**纯计数/状态投影**（无姓名/wxid/正文/标签）：`amr_version` / `killswitch` /
`autonomy{off,observe,supervised 会话计数}` / `auto_replies{armed,shadow,human}` /
`outbox{pending,failed_recent}` / `backends[]{slot,tool,reachable,backend_version}` /
`events_recent.errors_24h`。另有 `GET /api/version`（`amr_version` + `consumes` 消费清单）。

- **告警条件**（建议 60s 轮询）：`killswitch` 翻转 / 任一 `backends[].reachable==false`（连续≥2 轮）/
  `errors_24h` 超阈值（默认 >5）/ `outbox.pending` 卡住不降 / `amr_version` 与预期部署不符。
- **降级别误报**：`auto_replies==null` 或 `backends[].reachable==null` 是**指标本次降级**（探测慢/异常），
  **不代表 AMR 故障**，连续多轮 null 才告警。
- **人在回路红线**：监控 Agent **只观测+告警、不操作**。控制动作（killswitch / 拨挡 / outbox 确认发送）
  仍是**主令牌 `JL_WEB_TOKEN` + 人**的领地，**不把主令牌交给外部 Agent**。

### 2. 版本可见性 + 契约版本规范（两版本轴）

> 规范正本：`docs/superpowers/specs/2026-06-28-contract-versioning-and-compat-v1.md`。
> 单一真相源：`src/jl/version.py`（`__version__` + `CONSUMES`），与 `pyproject.toml` 由测试守不漂移。

**两条版本轴别混淆**：
- **后端软件版本**（如 fullwechat `backend_version`）—— 后端那台程序的版本。
- **契约版本**（如 `message.canonical/1`、`send-target/v1`）—— 双方约定的数据/接口契约版本。

```sh
jl account ls        # 看每个后端：tool / host / 后端软件版本 / 它 PROVIDE 的契约 schema
```

- **双向握手**：后端声明它 **PROVIDE** 什么（`/api/status.version` + `/api/capabilities.schema`）；
  AMR 声明它 **CONSUME** 什么（`version.py` 的 `CONSUMES` + `/api/version`）。
- **三方可见**：用户（UI 右下角徽标）/ 运维（`jl account ls`、`/api/version`）/ Agent
  （`/api/version` + 适配器每次出站带 `X-AMR-Version` / `X-AMR-Consumes` 头告诉后端）。
- **非对称是正常的**：后端可 EXPOSE 一个 AMR 还没 CONSUME 的契约（如 `group.canonical/1` 群名册，
  当前 `CONSUMES` 里为 `None`）——这种落差正是兼容矩阵要暴露的，不是 bug。

### 3. 接账号身份预检（防绑错号）

`jl account onboard`（配置驱动接 fullwechat 后端，见 `docs/runbooks/onboard-fullwechat-account.md`）
在 apply 前做**身份预检闸**：

- 拉后端 `/api/status/auth` 的 `loggedInUser`，**比对其 base 是否 == 注册表 `self_id`**。
- 不等 → reason=`mismatch`，**拒绝接入**（防止把 token/账号绑到错的号上）。
- 其它拒绝态：`no_token`（cred 文件缺）/ `unreachable`（后端不可达）/ `no_user`（响应无 loggedInUser）。
- 流程：dry-run（预检 + 打印计划）→ 人确认 → `--commit` 写 `accounts`。
- **cred 约定**：`~/.config/jl/cred/<tool>_<account_id>.token`，**chmod 600，带外交付，绝不入仓**。

### 4. 监管下自动回复——运维急停 + 挡位（killswitch）

v0.11.0 引入"监管下自动回复"（默认全关）。运维侧要会**急停**和**看挡位分布**：

- **全局 killswitch（急停一切自动发送）**：人在回路红线动作，走**主令牌 + 人**，外部监控 Agent 不可碰。
  `/api/health` 的 `killswitch:true` 即急停态；翻转即应告警。
- **看挡位分布**：`/api/health` 的 `autonomy{off,observe,supervised}` 是**纯会话计数**——
  `supervised` 突增 = 有更多会话被开了监管自动，留意。
- **看"已就绪待发"**：`auto_replies.armed > 0` = 有自动回复候选**已过双闸、在倒计时待人确认**，需人留意。
- **闸的语义**（排障时心里有数）：自动回只在 **双闸全过**（闸一=话术库白名单命中 ∧ 闸二=LLM 判低风险）
  **且** 时间窗（默认 9–21 点）**且** 未超人本限频时才会 arm；**实质/情绪回复一律交人**，**reply-only**（只回不主动撩）。

### 5. 发送可靠性排障

| 症状 | 原因 | 处理 |
|---|---|---|
| 发送超时 | 后端慢；超时已从 30s 放宽到 **60s** | 60s 内多能成；持续超时查后端负载/网络，别用更短超时自掐 |
| 后端返回 `opened:false`（开会话卡死） | **系统性**问题：后端开冷会话这一步卡住，不是偶发抖动 | **别无脑重试**——重试只会再卡。判为后端侧问题，**等后端修**；结合 `/api/health` 的 `backends[].reachable` 与 `errors_24h` 定位 |
| `backends[].reachable:false`（连续多轮） | 该后端掉线 | 查后端进程/网络/token；恢复前该账号无法收发，告警给人 |
| `outbox.failed_recent` / `errors_24h` 偏高 | 多为后端掉线或发送链路问题 | 结合 `backends[]` 排查；`errors_24h` 当前用「近 24h 失败 outbox 数」作代理指标 |

### v0.11.0 凭据 / 红线小结
- 接账号 cred：`~/.config/jl/cred/<tool>_<account_id>.token`，600，**带外，不入仓**。
- 主令牌 `JL_WEB_TOKEN`：控制面（含 killswitch / 拨挡 / 确认发送）专用，**不交外部监控 Agent**；
  外部 Agent 只给 `/api/health`（零 PII、无令牌）。
- 所有 token / SSH / 真实主机地址：**带外、不入公开仓库**，文档一律用 `<AMR_HOST>` 占位。
