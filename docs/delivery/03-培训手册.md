# AMR 培训手册（受众：现场 Build 工程师 + 数字运维工程师 Agent）

> 适用版本 **v0.10.0** ｜ 更新日 2026-06-28
> 写法：面向 **Agent + 工程师可直接照做**的首次交付上手流程。深度部署/排障见运维手册。

## 首次部署引导（PHASE 顺序）
1. **前置**：部署主机有 python3.10+、git；目标渠道后端可达（fullwechat REST、lark-cli 已鉴权）。
2. **拉代码 + 装服务**：`deploy/deploy.sh <user>@<host>`（首次自动生成 web_token + amr.env + 开 linger）。
3. **配方法内容（本地，不入库）**：放置 `~/.config/jl/playbook.md`、`diagnosis_rubric.md`、
   `persons.json`（种子）。缺则 AI 功能降级，不阻断。
4. **配 LLM（可选）**：以运维账号登录 `claude`（Max plan），amr.env 设
   `AMR_LLM_PROVIDER=claude_code` + `AMR_CLAUDE_BIN=<绝对路径>`。不配则纯人工模式可用。
5. **入库 + 拉数据**：`jl --migrate`（persons.json→DB），`amr-poll` 自动拉新；或 `jl ignite`。
6. **验收**：浏览器开 `http://<host>:8088/?token=<web_token>`，三栏可见、能开会话、能拟话术、
   outbox 能确认发送（先拿自有测试号验证真发）。

## v0.8 上手 delta（self 设置 + 归一，Agent/工程师照做）
> 在上面 6 步部署完成后，首次交付时多做这两步——都在 ⚙ 设置页或 CLI。

7. **自我身份声明（首启必做）**：
   - 自动种子：服务首启时已从 `accounts.self_id` 把运维/客户本人渠道身份纳入登记（`seed_self_from_accounts`）。
   - 人工补全：开 **⚙ 设置 → 自我身份**，系统列出**建议候选**（名字含「我/本人」或等于本人真名的
     会话 peer），**让客户本人勾选认领**（机器不擅判 self，不自动入）。每个身份打 persona
     面具，默认 **{自我 / AI分身 / 经营}**。
   - 验收：认领后，该本人发出的消息时间线显示为**「我」**（`apply_self_directions` 已把
     `sender_id ∈ self` 的消息标 `direction=out`）；自我会话归「🪞 我的」区，**不出现在
     「📞 该联系谁」/主动/开场**。
8. **首次归一**：
   - 启用自动归一开关后，首启自动跑一次 `seed_self_from_accounts` + `reunify`（全扫 + 强信号
     自动连 + 弱信号留候选）。
   - 人工：⚙ 设置 → 归一，点 **🔄 启动归一**，把归并候选（带电话/wxid 匹配证据）交客户确认；
     并错的用 **🔓 解绑**（`jl 解绑 <conv>`）拆开。
   - 等价 CLI 见运维手册（`jl 电话归一` / `jl 解绑` / `jl 连` / `jl 关注`）。

## 给营销专员的 5 分钟培训脚本（人机协同）
> 目标：让营销专员看界面即会（UI 自明），培训只需点一遍动线。
1. 左栏点「该联系谁」里的人 → 中栏看完整对话 → 右栏点「🩺 诊断」→「✨ 话术·用此版」→
   改 → 中栏点「发送 →」→ **原地**「✅ 确认发给 X」（v0.8 改为就地内联确认，不跨栏跳）。
2. 强调铁律：**AI 只给草稿，发不发你定**；断网也能手敲照常用。
3. 关注/记一件事/办结：把要主动跟的人「关注」，把要办的事「记一件事」、办完「办结」。

## 人 / Agent 分工
- **现场 Build 工程师**：部署、配渠道凭据、首次验收、培训交接。
- **数字运维工程师 Agent**：日常 poll 监控、电话/渠道归一、排障（见运维手册排障表）、
  方法内容(playbook/rubric)初稿维护（人审定）。
- **营销专员（人）**：日常用三栏收发、审话术、守承诺。
- **营销数字专员 Agent**：起草话术/诊断/主动 opener（全过人审闸）。

## v0.10.0 培训 delta（今日简报 + 可换皮肤 + 手机端）

> v0.10.0 最大变化：**默认落地页从三栏收件箱换成今日简报**。培训时需帮客户建立新的第一印象。

### 首次打开——新用户看到的是简报

v0.8 及以前，打开看到三栏收件箱；v0.10.0 起，打开看到**今日简报**（五张报告卡 + 「需你拍板」条带）。

**给客户的一句话定格框架**：
> "你是控制者，不是读屏的。简报帮你 Agent 摘好总结，你只管看一眼、拍板放行或改改。"

**讲解动线（5 分钟）**：
1. 带客户浏览五张报告卡——强调"数字是真实 SQL 统计，没有 AI 也能看"。
2. 带客户点一次「需你拍板（N 项）」——点「✅ 放行」一次，解释"放行 ≠ 自动发出，消息仍需在对话里再点确认发给 TA"。
3. 从关系报告卡点进去，进入 L1 人列表，再点一个人进 L2 会话（三层钻取动线）。
4. 演示切皮肤：右下角切换器 → 选「收件箱」→ 还原旧三栏；再切回「简报」。告知"两套皮肤用的是同一份数据"。
5. （如有手机）展示手机底部 tab：📋简报 / 👥人 / 🗂事 / ⚙我，单列卡片流，操作逻辑同桌面。

**注意事项**：
- 「待后端」占位卡（如营销线索维度）解释为"后端还没接入，不是没数据"，不要让客户误以为系统报错。
- 简报数据**零 LLM** 跑通；有 LLM 时只是在数字下方加叙事总结。**无模型时简报正常显示**，客户不用担心"AI 挂了没法用"。

### 7 步部署后的验收补充（v0.10.0）

在原验收步骤 6 之后，额外验收：
- `GET http://<host>:8088/api/digest?token=<web_token>` 返回 200 JSON（简报数据接口）。
- 浏览器默认落地页为今日简报（非旧收件箱）。
- 右下角皮肤切换器可见；切 `inbox` 可进旧三栏；localStorage `amr_skin` 值为 `digest` 或 `inbox`。
- 手机（或窄屏浏览器）打开，出现底部 tab 栏、单列卡片流。
