## §0 这份文档是什么（三个受众）

把 AMR / AMP / fullwechat 开发这套微信 Agent 体系时**反复遵循的底层原则**，提炼成**一份开发宪法**。三个用途：

1. **内部效率** —— 任何新项目、新 Agent **开工前第一件事读它**，统一遵守 → 提速、质量一致。
2. **团队对齐** —— AMR / AMP / fullwechat 各 Agent 同一套原则，不各自漂移。
3. **⭐ 兼爱书院教学课程** —— 教想做程序的人**都能开出自己的好产品**（服务于人的 Agent 软件）。所以本文**写成可教学的课程料**：每条**能讲、能背、能上手**，不只是内部备忘。

## §1 怎么用

- **新项目开工**：在该项目 `CLAUDE.md` / `AGENTS.md` 第一行引用「开工先读本文件」。
- **拿不准时**：回到原则。原则冲突时，**§2 思维框架 + HITL 最高准则**优先裁。
- **有异议**：互相提、迭代版本号，不私自背离。

## §2 格式约定（每条原则）

> **原则名** ——「一句话」。**为什么**：…。**正例 / 反例**：…。**红线**（适用时）：…。**证据**：仓里哪个文件/特性。

教学口径：先讲**为什么**（建心智模型），再给**正反例**（可迁移），最后**红线**（不可越）。

## 原则 0 · META — 用「四把尺子」理解每个技术决策（贯穿全篇）

> 王总思维框架，是讲解所有原则的**镜头**，不是单独一条。

- **总分总**：先给宏观结论 → 拆分点 → 收回宏观。讲原则、写设计、提交说明都套这骨架。
- **多角度跨学科**：时间/空间 × 竞争/合作 × 人性 × 经济政治地理历史数理化生心理医健 —— 一个决策从多维看。
- **物质·信息·能量**：任何系统/企业经营都是这三者的转换。软件主要搬运**信息**，但要算它消耗的**能量**（算力/人力）与沉淀的**物质**（数据资产）。
- **算力 × 算料 × 算法**（IT 演进史纵轴）：看任何选型，问这三者如何**相互兼容、彼此 trade-off、一步步推动演进**。
  - 例：AMR「双闸」为什么闸一是确定性话术库、闸二才是 LLM？因为**算法(LLM)不稳定且费算力**，先用**算料(人灌的白名单)**做确定性地板，把 LLM 降为"只能否决"的二道闸 —— 这是算力/算料/算法的折中。

**教学法**：讲任何一条原则，都用这四把尺子至少一把解释"为什么历史/系统会演化成这样"。

# Part A · AMR 侧原则（本文写全）

## A1 · 人·事双轴模型 —— 不是聊天软件，是把每件事办到闭环的工作台

「一个界面，左看**人**、右看**事**；事会**自己冒出来**，不靠人记。」

- **为什么**：传统 CRM 的业务记录是**死的**，人一忙就烂在那被忘掉。把「事(matter)」做成**引擎**：承诺到期 / 事闲置 → **主动浮出**要人处理。人(person)与事(matter)是 **M:N**（一件事挂多人、一个人多件事）。
- **正例**：`lifecycle.propose` 扫出"承诺未结/闲置"信号，简报里冒成"待办由头"。**反例**：把对话记完就完事，靠人脑记谁欠谁什么。
- **红线**：事的状态流转（open/handled/dropped）留痕，谁/何时/为什么，不黑箱。
- **证据**：`src/jl/lifecycle.py`、设计 spec §1.5/§2、dual 皮肤（人视图/事视图）。

## A2 · 端点一等公民 —— 同一个人散在五个口子，系统替你挑当下最通的那个

「路由的是**通道与关系**，不是盲发消息。端点(endpoint)是头等对象。」

- **为什么**：一个人有好几个微信号 + 电话 + 飞书。发到一个不看的旧号 = 丢单。按 **weight × recency + pin** 选**当下发得出去**的端点；号码去重；可钉首选。更深一层：**同一渠道有多种接入工具**（fullwechat / PowerData），按**账号绑定工具 + 能力路由** —— 这是 Router 的命门。
- **正例**：`routing.best_endpoint`（pin 覆盖、不可达降级）；per-account fullwechat 轮巡。**反例**：渠道盲发、把"微信"当一个端点。
- **证据**：`routing.best_endpoint`、`channels.pinned`、记忆 `channel-vs-access-tool-router-trait`。

## A3 · 监管下的 agentic router = 纵深防御 —— 让"AI 自动沟通"敢交给它

「**观察 → 监管 → 自治**三挡由人手动拨；**双闸 + 时间窗 + 人本限频 + 全局刹车**四层纵深；系统**绝不自作主张升挡**。」

- **为什么**：客户最大的怕 = AI 乱发/发错/半夜扰人。纵深防御逐层拦：
  - **闸一（确定性）**：命中**人灌的话术库白名单**才可能自动 —— 零 LLM、可解释。
  - **闸二（LLM 风险低）**：LLM **只能向人 veto**（判高风险→交人），**永不越白名单授权**。判不准/模型挂 → 交人。
  - **时间窗**：上班时段才发，不半夜扰人。**人本限频**：模仿真人能照顾的量，不刷屏。
  - **全局 killswitch + 人占闸**：随时一键全停；人在自己号上操作时 Agent 即冻结（见控制权契约）。
- **正例**：`gate.classify`（gate1∧gate2）+ `autocomms.propose_replies`（propose-only，真发只走 outbox/confirm 倒计时否决窗）。**反例**：让 LLM 直接决定发什么、自动群发。
- **红线**：**LLM 只做"向人 veto"，永不"替人授权"超出白名单**；广播(朋友圈)永在自动闸之外、永远人审；挡位只能人往上拨。
- **证据**：`src/jl/gate.py`、`src/jl/autocomms.py`、设计 spec §7.5、Phase-1 实现。

## A4 · 记忆层 recall / expand —— 拟话术前先想起这个人

「给某人出话术前，**先 recall** TA 的上下文（近况/未结的事/到期承诺/关系温度），再拟。」

- **为什么**：脱离上下文的话术是套话。一个**预算受控的检索**把"该记得的"捞回来；每个结果给 `handle` 可 `expand` 深挖。**纯 SQL、零 LLM** —— 记忆不依赖模型。
- **正例**：`recall.recall(conn, person_id, budget=8)` → recent/open_matters/due_commitments/temperature。**反例**：每次把全量历史塞进 prompt（费算力、还更糊）。
- **证据**：`src/jl/recall.py`。

## A5 · SELF 归一 —— 连你自己都认得，名单里只剩真客户

「用户的多渠道多身份 / 多 persona 面具，**归一为"我"**；自己发的显示为"我"，不混进"该联系谁"。」

- **为什么**：人在不同渠道有多个号、多重身份（昵称/真名/身份证名/商务尊称）。系统若不认得"这都是同一个我"，会把"我和我分身的对话"当成客户、把自己排进待联络队列。
- **正例**：自我身份认领 UI（勾选认领本人多号）→ 自己的消息标"我"。**反例**：把用户本人的另一个号当陌生客户去"撩"。
- **证据**：记忆 `self-identity-and-personas`、自我身份认领特性。

## A6 · 主动调度 + outbox 确认地板 —— 把杂活交给数字员工，把舵握在自己手里

「**主动发起 = Agent（数字员工）**；**决策权 = 留在人**。真发**只走 outbox → confirm 否决窗**，系统**永不替你发**。」

- **为什么**：AI 时代人不被替代 —— 人做**创造与决策**，Agent **放大与执行**。Agent 可主动浮出"该联络谁"、拟好 opener，但**发不发永远人点**。一切对外写收口到 outbox/confirm 一个地板，可留痕、可否决。
- **正例**：`assist.proactive_sweep` 出 opener 草稿进队列；`/api/outbox` → `/api/outbox/confirm` + 倒计时否决。**反例**：Agent 拟完直接发、fire-and-forget。
- **红线**：**没有任何代码路径绕过 outbox/confirm 发出对外消息**；草稿再像也只是草稿。
- **证据**：`assist.proactive_sweep`、`/api/outbox(/confirm)`、countdown 否决窗。

## A7 · 契约即真相源 —— 跨系统先立约，谁拥有谁定，后端不自行解释

「跨系统接口**先立契约、单一真相源**：AMR 定契约，fullwechat 实现；有疑义**提 AMR 改 spec**，后端不硬编码推断。」

- **为什么**：两个系统各自猜口径 → 行为漂移、对不上。把口径写进**一份 spec**，谁拥有谁定（消息/发送/群名册/朋友圈/控制权都在 AMR 仓）。契约**先于实现**，能力声明 + 优雅降级。
- **正例**：5 份契约（message-canonical / send-target / group-metadata / moments v1+v2 / control-and-voice），每份"后端遇不清楚→提 AMR 改 spec"。**反例**：后端自己解释字段、硬编码一套，AMR 再适配另一套。
- **红线**：后端**不自行解释契约、不硬编码推断口径**；有歧义回问真相源。
- **版本即契约的一部分**：契约有版本，消费方也有版本。两侧版本**双向声明、双向可读、可对账**——见 `2026-06-28-contract-versioning-and-compat-v1.md`（契约版本轴 / 后端软件版本轴 / AMR 消费侧版本 + 消费清单 + live 兼容矩阵）。属**运维手册**给 FDE 的对账依据。
- **证据**：`docs/superpowers/specs/2026-06-*-*-contract*.md`。

## A8 · LLM-optional 地板 —— 模型全挂，人仍在回路、仍能沟通

「读 / 搜 / 浏览 / 回 / 发 —— **零 LLM 也照常**。LLM 是 assist，不是 gate。」

- **为什么**：把模型当 gate，模型一挂业务就停。把它降为**可选 assist 层**（草稿/摘要/语义检索/ASR/自动路由），不可用就优雅降级到手动 —— 这是 HITL 推到极致：人能脱离任何 AI 独立运转。多 LLM 走**薄抽象 + 按任务/成本/可用性路由**，不硬编码单一供应商。
- **正例**：`llm.available()` 处处降级；染色/路由/recall 全是确定性 SQL。**反例**：诊断/话术不可用就整个收件箱瘫痪。
- **红线**：**任何人类沟通能力都不得以"模型可用"为前置**。
- **证据**：CLAUDE.md 钦定、`src/jl/llm.py`（provider-agnostic + 降级）。

# Part B · fullwechat 侧 18 条（仁德@ReleaseWX 合入此区）

> 与 Part A 重叠的合成一条（见去重表）：B1→A7、B2→原则-1总纲、B3→A8、B13→A6、B18→原则0。下列为 fullwechat **保留条目**的完整教学正文。

## B4 · 稳定 id 为锚，绝不模糊误发 —— 一次发错，信任就崩
「用不变 id(wxid/chat_id/localId)寻址 + exact 验证，**绝不按显示名模糊匹配**。解析不到唯一目标 → 报错不发。」
- **为什么**：一次误发/误操作=信任崩塌+风控+不可逆。宁 Failed 不瞎发。同名极多（搜 connie 出一堆），名字会变、id 不变。
- **正例**：冷会话搜索后**用 exact chat_id 重试 chat-select 验证**才发；群成员按 username 精确 join。**反例**：搜出 top result 直接发、按昵称猜一个。
- **红线**：多匹配/解析不到 → `ok:false`，**绝不"猜一个发出去"**。
- **证据**：`chat_search.rs`(exact-key 重试)、`wechat_groups.rs`(username join)。

## B5 · 真机验证 / 吃自己狗粮 —— 对着现实跑，不靠假设
「自己/小号先在真机跑通，再复制给客户；字段对着真机字节解，不推测。」
- **为什么**：a11y 树/protobuf 字段/风控行为，假设十有八九错。易链先用(dogfood)，验通才上莱妈/张总/thfs。
- **正例**：群 roster protobuf 对着真机 roomdata 逐字节解码(field2=群昵称、field4=邀请人)，改完真机复核 5/5、11/11、464/464。**反例**：照"常见布局"猜字段号 → roster 全错位。
- **证据**：roster 真机校准、codebandi dogfood、易链优先。

## B6 · 三接口都暴露（CLI / REST / 程序内）—— 同能力，三个入口
「同一能力同时给命令行 / HTTP / 程序调用，不锁死在一个渠道。」
- **为什么**：运维要 CLI、AMR/AMP 要 REST、FSM 要程序内；锁一个就堵死另两类消费者。
- **正例**：点赞 = `pdwx moments like` + `POST /api/moments/{tid}/like` + `LikeMomentPlan`。**反例**：只给 REST，运维想脚本化没门。

## B7 · 可观测 + 运维即竞争力 —— 看得见才管得动，长期运维是壁垒
「健康/日志/告警都**机器可读(JSON)**，给人也给 Agent；远程运维 + 长期服务做成**产品竞争壁垒**，不是附属。」
- **为什么**：黑箱出事才知道。结构化可观测让数字运维 Agent 7×24 巡检 + FDE 兜底 = 续费点。
- **正例**：`/api/health/deep` + 结构化 event log(全哈希) + `/api/events` + 飞书告警 poller。**反例**：只 `docker logs` 被 x11vnc 噪声淹没、`/health` 只回 200。
- **证据**：ops-agent-contract、运维三件、event_log.rs。

## B7.5 · 开发期就内建日志/Debug —— 看得见才改得动，盲调是最大浪费 ⭐
「开发任何 Agent 动作/FSM，**第一件事就埋阶段日志 + Debug 信息并暴露出来**(API/事件流/文件)。开发期可观测 = 有效开发的兜底；运行期 ops 日志(B7)是它的延伸，开发/运维一套可观测两用。」
- **为什么(血泪)**：本项目朋友圈点赞调了一轮又一轮全在猜(改点击/popover/窗口激活/几何)，直到给 `moment-like` 加一行日志才发现 **"plan 根本没调到它"——真 bug 在上游，前面全白改**。没有阶段日志 = 盲人摸象 = 最大的时间浪费。
- **正例**：FSM 每个 phase 转换 `emit` 事件(Opening→ScrollToTop→Locate→EnsureVisible→Click→Verify)，`/api/events` 可读 → 一眼看出在哪 bail、传了什么参数、报了什么错。**反例**：plan 黑箱跑，只返回 success/error，出错只能逐层猜。
- **红线**：**新功能/新 FSM 不带阶段日志不算完**；调试前先加可观测，别空改代码。
- **算力×算料×算法 视角**：日志=最便宜的"算料"，用一点存储/IO 换巨量调试算力(人/Agent 的时间)——这笔 trade-off 永远划算。

## B7.6 · 测试驱动开发(TDD) —— 先写测试，用例即规格 ⭐
「写实现前先写测试：测试先红 → 实现转绿 → 重构。测试 = 可执行的规格 + 回归网。改 bug 先写复现它的测试再修。」
- **为什么**：没测试的代码改一处崩三处。本项目改 canonical/group 时靠 25 个单测兜住回归；roster protobuf 校准时**先加 `inviter-field4` 回归测试**才敢动解析。
- **正例**：`wechat_groups.rs` 7 个单测(roomdata 解析/去重/wxid 启发式/邀请人不当昵称)，`cargo test wechat_groups` 把关。**反例**：写完手动跑一遍就提交，下次改动悄悄破。
- **红线**：核心逻辑(解析/映射/契约实现/id 解析)**无测试不算完**；修 bug **先红后绿**。
- **GUI/FSM 例外口径(重要)**：纯逻辑(protobuf/canonical/id 解析)→单测;**GUI/FSM 真机行为(a11y 点击/弹窗)难单测**→用**真机验证(B5)+阶段日志(B7.5)** 兜底替代——这正是朋友圈点赞 FSM 卡很久的原因:不可单测,只能靠真机+日志逼出真相。二者配合,不留盲区。
- **补的测试工具**：`cargo test`(单测) + 真机验证脚本(GUI/FSM) + 合成数据夹具(契约/解析) + (待补)集成测试。

## B8 · 产品化 / 配置驱动 —— 造一次，N 客户复用
「不做一次性脚本；一套系统读配置服务多客户。」
- **为什么**：每客户重写=边际成本高、不可规模化。配置驱动→边际成本趋零。
- **正例**：运维 poller 读 `ops-targets`(每客户:实例+运维群)；命名规范一键生成；易链/莱妈/张总/thfs/一晶钻头同模式。**反例**：给易链硬编一套、莱妈再抄一套。

## B9 · 边交付边固化（交付即开发）—— 踩的坑当场沉进产品
「交付客户即打磨产品，把学到的**当场沉进套件/文档/代码**，不二分"开发"和"运维"。」
- **为什么**：王总"生产就是开发"。一次性修不沉淀=下个客户再踩。
- **正例**：每个真机坑(noVNC LAN/命名做旧/scroll坐标)当场修进 deploy kit + 运维手册。**反例**：手动救火完不留痕。

## B10 · KISS / 复用不重造 —— 少发明轮子
「最大化用现成工具、已验证逻辑；能复用绝不重写。」
- **正例**：复用 media-export 解密、scroll/click 工具、contact.db 读取栈。**反例**：为群读重写一套 SQLCipher 开库。

## B11 · 拟人化 / 避风控 —— 别做完美机器
「慢、有节奏、有失败率，像人；设备名做旧、禁含 wechat。」
- **为什么**：机器秒级完美=风控靶子。真人活动反而让账号更可信(呼应控制权契约"人手动用增价值")。
- **正例**：点赞随机停顿 + `MOMENT_LIKE_HUMAN_MISS_RATE`；hostname `Desktop-loong-2年前日期`。**红线**：**设备名禁含 wechat/微信**(否则拒登)。

## B12 · 不落 PII / 合规（两侧共守）—— 哈希，公开仓干净
「不打印凭据(token 不进 transcript)；公开仓无真名/wxid 明文/正文；日志全哈希。」
- **红线**：event log 只记 kind/计数/哈希/错误码，**绝不落正文/真名/wxid 明文**。
- **证据**：fullwechat hash_id；AMR 合成数据 + secrets-scan 钩子。

## B14 · 以终为始 / 受众驱动 —— 卖给谁就写给谁，无受众不写
「每份产出锁定**真实受众**；同一事实按受众改写。」
- **正例**：4 套手册各锁一受众(销售/营销专员/现场工程师/运维 Agent)，营销资料讲 ROI 不讲技术。**反例**：一份糊给所有人。

## B15 · 业务语义分层：Marketing vs Selling —— 先问这是哪类工作
「按业务功能给 Agent 动作/工具归类：**Marketing**(广播·获客·内容) vs **Selling**(一对一·关系·成交·服务)。」
- **为什么**：同一场景(朋友圈/千牛)拆成两类工作→工具各司其职、产品理念清晰、可复用到新场景。
- **正例**：发朋友圈/上产品/营销资料=Marketing；点赞/评论/接待客户成交=Selling。**用法**：围绕任何新场景开发先问"Marketing 还是 Selling"再定形态。

## B16 · 培训是交付一环 —— 工具好用，人得会用
「Agent 要"用得好"，人必须被培训(Agent 让人休息，但 onboarding 必须)。」
- **正例**：交付含培训手册 + 现场上手引导。**反例**：丢个工具不培训=白配。

## B17 · 自明性（Self-Evidence）—— 人只管用 + 决策，不管配置
「手册自明、UI 自明、流程自明；**尽可能少配置**。」
- **正例**：一条命令 `deploy.sh`；`login.sh` 自动弹 QR；`ops-check.sh` 即开即用。**反例**：交付前要客户填一堆 yaml。
