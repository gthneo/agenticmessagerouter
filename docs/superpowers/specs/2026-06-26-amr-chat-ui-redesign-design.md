# AMR 聊天界面重设计 — 设计 (spec)

> 日期 2026-06-26。范围：收件箱 Web UI 的聊天阅读与发送交互重设计 + 移动端响应式。
> 全部改动落在 `src/jl/web.py` 的 `INDEX_HTML`（CSS + JS）；后端 send/outbox/审计路径不动。
> 经两轮 brainstorm 收敛（王总 2026-06-26 钦定关键语义）。

## 目标 / 动机

当前收件箱中栏是平铺消息列表，阅读体验弱；发送交互（发送→确认发给X→改改）累赘且易误触、可重复点；不适配手机小屏。本次重设计三件事：**气泡式聊天（仿微信）+ 统一倒数发送闸 + 移动端两级响应式**，并借此把"人在回路"的交互逻辑理顺、不给人错觉。

## 核心原则：重定义「系统永不替你发」

**人在回路 = 人做决策**（从 N 条话术里选 1 条 / 定方向 / 随手插话），**Agent 执行所选**；倒数窗口是**否决窗**，人随时能拦。这不是"每条都手点发送"，而是"人当决策者、Agent 当执行者，且人始终能否决/接管"。

**三态视觉永不混淆**（防错觉的硬约束）：
- **真实已收**：中栏左侧浅灰气泡（带发言人名）。
- **真实已发**：中栏右侧绿色气泡。
- **AI 草稿**：右栏卡片，**不是气泡**。
- **待发（倒数中）**：输入区上方的**倒数条**，**绝不在气泡流里伪装成已发气泡**；倒数结束自动发后，才作为右绿真实气泡出现。

## 组件设计

### 1. 气泡聊天流（中栏 / 移动端聊天页）
- 对方消息：左对齐浅灰气泡，气泡上方小字显示发言人名（群聊/多账号下有用）。
- 自己消息（`direction='out'`）：右对齐绿色气泡。
- 连续同一发言人的多条合并视觉分组；按时间间隔（如 >5 分钟）插入居中浅灰**时间分隔**。
- **系统消息**（撤回/拍一拍等 `clean_content` 已转可读文本者）：居中灰色小条（如「"〇〇" 撤回了一条消息」）。
- 媒体/链接：显示 `clean_content` 已清洗的文本（`[图片]` / `[链接] 标题` / 文件名…），长 URL 自动换行（已有 `.m div{overflow-wrap}`）。
- 不做头像（无头像数据源）；以名字 + 气泡承载，保持轻量。

### 2. 统一倒数发送闸
触发：人**选中右栏某条话术（用此版）** 或 **在输入框自己打字后点「发送」**——两条入口都进**同一条倒数条**。

倒数条（浮于输入区上方，醒目）：
```
⏳ <N> 秒后自动发给 <对方>   [立即发]  [改改]
[ 待发正文（可见全文）…… ]
```
- 含一个随秒数收缩的进度指示（视觉强化"会自动发"，防错觉）。
- **焦点默认落在「改改」**：避免误触 Enter 直接发（保守默认）。
- 倒数到 0 未被打断 → 自动发 → 变为右绿真实气泡。
- **「改改」** = 取消倒数 + 焦点进输入框（编辑/改写/自己写）。
- **「立即发」** = 立刻发送，并**立即置灰禁用**该按钮（防多次点造成重复发送）。
- 发送实现仍走既有 `POST /api/outbox` → `POST /api/outbox/confirm`：**留痕（events kind=send）不变**，倒数只是 N 秒后自动调用 `confirmSend()`。

设置项（`⚙ 设置` 面板，存 `localStorage`，纯 UI 偏好不入库）：
- `自动发 [开/关]`（默认 **开**）。关掉 → 倒数条退化为「必须点 `[确认发]` 才发」的严格模式（回到旧铁律，谨慎用户/场景的退路）。
- `倒数 [N] 秒`（默认 **5**，整数，下限 1）。

### 3. 移动端响应式（微信式两级）
- CSS `@media (max-width:640px)`：三栏 flex → 单视图，由一个轻量「视图状态」(list / chat) 驱动显隐。
- **列表页**：顶部 `会话 / 人 / 事` 段切换；下方对应列表（复用现有 该联系谁/联系人/会话/outbox/候选 区块）。
- 点会话 → **全屏聊天页**：顶部 `← <对方>` 返回 + `🩺事` 入口；中部气泡流；底部输入 + 话术建议条（可展开 N 版）+ 倒数条。
- `🩺事`（matters/诊断/承诺）做成聊天页内**上滑面板**。
- 桌面端（>640px）维持现三栏布局不变；新气泡样式与倒数闸两端通用。

## 数据流 / 不变量
- 后端 API 与数据库**零改动**：复用 `/api/conversations/{id}/messages`、`/api/outbox`、`/api/outbox/confirm`、`/api/conversations/{id}/suggestions` 等。
- HITL 审计不变：每次真实发送仍写 `events(kind=send, actor, ts, detail)`。倒数自动发 = 自动触发 confirm，仍留痕。
- 自动发开关/秒数是**客户端 UI 偏好**（localStorage），不影响后端语义；关掉即严格手动确认。

## 范围（YAGNI / 不跳级）
**本轮做**：气泡界面 + 单条倒数发送闸 + 移动端两级响应式。
**本轮不做**（留下一阶段 **1c 渐进自治**，单独 spec）：对某人「锁定风格 → Agent 自动起草并连发后续多条」的自治推进。本轮每条消息仍经「人选 1 条 / 人打字」这一决策点。

## 错误处理 / 边界
- 倒数中切换会话 / 点其他操作 → 取消当前倒数（不串发到别的会话）。
- `立即发` 或自动发失败（send 返回 error）→ 倒数条变红显示错误，正文回填输入框，不置灰，便于重试。
- `自动发=关` 时无倒数，按钮为 `[确认发]`，点击即走 confirm（保持防重复置灰）。
- 自己打字为空 → 不触发倒数。

## 测试 / 验证
- 纯 `INDEX_HTML`（CSS+JS）改动，无 Python 单测；后端未改，现有 276 测试应保持全绿（回归保护）。
- 浏览器集成验证（Claude-in-Chrome）以一条真实会话为例：
  1. 桌面：气泡左右分明、系统消息居中、长 URL 换行；左栏不消失（已修）。
  2. 倒数闸：选话术→倒数→自动发→变右绿气泡；焦点在「改改」；改改→聚焦输入框；立即发→置灰防重复；设置关自动发→退化为手点确认。
  3. 移动视口（≈390px）：列表↔全屏聊天两级导航；事/话术面板可达。

## 风险
- 倒数自动发偏离"主动确认"铁律——以**显著倒数条 + 焦点在改改 + 全局可关**三重设计守住"人可否决/可退回"，并文档化新语义。
- localStorage 偏好不跨设备——可接受（UI 偏好）；未来若要跨设备再迁服务端。
