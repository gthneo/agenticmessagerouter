# AMR v0.9 — 可扩展多工具消费层 + PowerData 适配器 — Design / Contract

**Date:** 2026-06-16 ｜ **Status:** approved (brainstorm 收敛, dogfood 验证) ｜ 受
[[channel-vs-access-tool-router-trait]] 约束。本文件兼作**两个并行 agent 的共享契约**。

## 验证依据（为什么必须做，dogfood 实证；真名/真 id 留本地、不入库）
真机用 PowerData 读到：**一条高风险危机会话只存在于"只能用某一工具读"的自有账号上**，
另一工具(fullwechat/.178)完全看不到 → 不接多工具，AMR 对关键会话是瞎的。另：**同一个联系人
(同一个 wxid)在用户的两个自有账号上都可达**(一个热、一个冷) → 必须按稳定 id 跨账号/工具归一。

## 核心：渠道 × 接入工具
渠道(channel)=媒介(wechat…)；**接入工具(tool)**=AMR 怎么读/发该渠道。一渠道多工具：
wechat 有 **fullwechat**(读+发) 与 **powerdata**(只读)。账号绑定到工具；工具声明能力。

## Block A — 可扩展多工具消费层（owns: db.py / send.py / ingest_run.py / tests）
1. **schema/db**: `accounts` 加 `tool TEXT NOT NULL DEFAULT ''`（哪个工具服务此账号）。
   `_ADDED_COLUMNS` 加 `accounts:[("tool","TEXT NOT NULL DEFAULT ''")]`（旧库幂等迁移）。
   `upsert_account(... tool="")` 接受并存 tool。
2. **工具能力契约**：`IngestAdapter` 子类带类属性 `platform`(已有) + `tool`(str) + `can_send`(bool)。
   现有适配器补默认：fullwechat `tool="fullwechat" can_send=True`；lark `tool="lark-cli" can_send=True`；
   phone `tool="callhistory" can_send=False`。（A 改这三个类的类属性即可，不改逻辑。）
3. **send 能力路由**：`send.send_message(platform, chat_id, body, *, tool=None)` —— 若给 tool 且其
   `can_send` False → 返回 `(False, "该工具只读，请人手发/换可发工具")`，**不外发**。SENDERS 注册表
   按 tool 键（保留按 platform 的向后兼容默认）。能力查询 `send.can_send(tool)`。
4. **不破坏现有**：默认 tool 为空时沿用现状（fullwechat/feishu 按 platform 发）。

测试(test_db/test_send)：accounts.tool 往返 + 迁移；send 对 can_send=False 工具拒发并给可读错。

## Block B — PowerDataAdapter（owns: src/jl/channels/powerdata.py + tests/test_powerdata.py，全新文件）
只读微信工具，读那些只能经它访问的自有账号。`class PowerDataAdapter(IngestAdapter)`：
`platform="wechat"`, `tool="powerdata"`, `can_send=False`。
- **传输**：MCP-over-HTTP JSON-RPC，POST 到 `POWERDATA_URL`（env / `~/.config/jl/powerdata_url`，
  **URL 与 token 都不入库**）+ `Authorization: Bearer <token>`
  （token 从 env `WX_MCP_TOKEN` 否则 `~/.config/jl/wechat_mcp_token`）。
  `_call(tool_name, **args)`：JSON-RPC `{"jsonrpc":"2.0","id":N,"method":"tools/call",
  "params":{"name":tool_name,"arguments":args}}`；MCP HTTP 需先 `initialize` 握手 + 复用
  `Mcp-Session-Id` 响应头；响应可能是 SSE(`text/event-stream`，逐行 `data: {json}`)或 JSON，
  两种都解析；取 `result.content[0].text`（即 prose 字符串）。**网络层尽力而为**，失败抛由调用方降级。
- **prose 解析（纯函数，重点单测）**：
  `parse_sessions(text)` → [{name, is_group, unread, preview}]（`get_recent_sessions` 输出）。
  `parse_history(text)` → [MsgRecord]（`get_chat_history` 输出行 `[YYYY-MM-DD HH:MM] 发件人: 内容`）：
  `ts`=该时间→unix；`sender`=发件人(>"me" 视作出站候选)；`content`=内容(已是 `[链接]/[图片]` 占位)；
  `msg_key=ingest.msg_key(source="powerdata", stable_id=sha1(f"{ts}|{sender}|{content}")[:16])`
  （PowerData 无 msg_id → 合成稳定键去重）；`direction="in"`（出站由 db.apply_self_directions 后处理；
  但发件人字面=="me" 的可直接标 "out"）。
- **IngestAdapter 方法**：`list_conversations(account)` → `_call("get_recent_sessions", account=…)`
  → parse_sessions → ConvRecord（群 muted=True；chat_id 暂用 name，见下注）；
  `pull_new(account, recent_limit)` → 每会话 `_call("get_chat_history", chat_name=…, limit=…)`
  → parse_history。`backfill` 同理带 offset。`send` → 返回 `(False,"powerdata 只读")`。
  **chat_id 注**：PowerData 按 name 寻址，无 wxid 直出；本批 chat_id 用联系人 name（备注/昵称），
  后续可用 `get_contacts` 补 wxid 做跨工具归一（留 TODO，不阻断本批）。
- 测试(test_powerdata)：parse_sessions / parse_history（合成 prose，断言 ts/sender/content/msg_key
  稳定 + 群识别 + 媒体占位 + me→out）；`can_send is False`；`send()` 返回拒发。**不测 live 网络**。

## 安全/原则
token 仅 env/本地文件、绝不入库；powerdata 只读 → 发送降级到"人手发"(守 HITL)；prose 解析合成
稳定 msg_key 保证幂等；LLM 无关。两 agent 文件不重叠(A: db/send/ingest_run/对应测试；B: 新建
powerdata.py + 新测试) → 可并行无冲突。两 agent 都**不 commit**(控制器统一提交)。
