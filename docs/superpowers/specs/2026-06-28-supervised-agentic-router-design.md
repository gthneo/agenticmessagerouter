# 监管下的人·事 Agentic Router — 架构设计

> 设计源:2026-06-28 王总(班迪)× 仁德(AMR)六轮探讨质询定稿。
> 本文是**设计正本**,不是实现。落地拆成多份 implementation plan(见 §12)。
> 最高约束不变:**LLM-optional core + 人在回路(HITL)= 最高准则**。

**目标**:把 AMR 从"消息路由器"升级为"**人·事 agentic router**"——在人的监管下,系统替你盯着每件业务的生命周期、按显著记忆主动撩客户/家人,出安全区就停下来交还人。

**一句话架构**:人/事是同一份消息 timeline 上的两根轴;Router 维护"事-归属";事是带生命周期的工单;生命周期引擎按事件双源提议、人确认地推进;推进的副产物是"撩由头";记忆层按显著性把"此刻该记得的"取出来喂双闸;双闸(确定性白名单 ∧ LLM风险)决定能不能在时间窗内自动发。

---

## §1 核心心智模型:两轴 + 一个难映射

三个实体、Router 维护两条映射:

- **人 ↔ 消息** — 几乎免费(通道身份天然知道谁发的)。
- **事 ↔ 消息** — **难的那条**,要靠解读;一条消息"属于哪件事"由 LLM-assist 提候选、人确认。这是 Router 真正的路由难题(**事-归属 attachment**)。
- **人 ↔ 事** — 经由消息派生:人通过"碰到这件事的那些消息/会话"连到事。

**关键认知:事横切 session,不是 session 的一段。** 一件事跨会话、跨通道、隔周重启(微信聊一半、电话补一刀、两周后再续)。故事是**盖在消息流之上、M:N 挂消息的 overlay**,不能用"会话片段"建模。

---

## §1.5 群组 = 场(arena):人↔通道 1:1 的破例

群组打破"人 ↔ 通道 1:1"根假设:一条群通道塞多个关系账户,分属 2-3 个组织。引入第三结构:

- **群 = 场(arena)**:头等实体(会话场),M:N 挂成员;**每个成员仍是独立关系账户、跨通道归一**(群里的张三 = 私聊/电话里的张三)。撩/跟进常瞄准群内某个决策人、且可能走 DM(另一通道),故"群"与"群内每人的关系账户"分开建模。
- **阵营/组织**:先做 person 的"所属组织"标签 + 群内阵营分组视图,**不建完整 org 实体**(YAGNI;除非要按组织聚合分析)。
- **多方事不用新模型**:§2 的事已 M:N 挂 persons;群里那桩多方谈的生意,只是挂的 persons 跨群+跨组织。

**群按"规模×目的"分两形态,串成一条营销漏斗:**

```
大营销群(20–500人,几人主讲,广播) ── 事 = 营销/传播事(type B)
   └ 浮出销售线索(谁关注/互动/提问 → 染色·信号)
   └ 线索转化(系统提议+人确认)→ 一键建小群场 + 落地事 + 带该人关系账户&历史
        ▼
小群(2–3组织利益相干人,1群1事) ── 事 = 落地业务事(type A)
   └ 生命周期推进(候选→进行→等待→完结=成交)
```

- **小群 ≈ 1 群 1 事**:群是"多方在场的容器",直接套 §2–§8。
- **大群 = 线索源/营销场**:不塞进日常人视图(噪声);UI 给"浮出的线索"列表 + 一键拉小群。
- **线索转化 = 系统提议 + 人确认**:确定性信号(互动频次/关键词)无 LLM 也跑,LLM 加语义线索分;识别"关注的人"→ 提议拉小群 → 人确认 → 一键建小群场+落地事+带历史。复用 §3 双源提议+人确认。**系统不自动拉群**(拉群=对外动作,HITL 守)。

---

## §2 事(matter)= 生命周期工单

事是**头等对象**,带生命周期 stage,M:N 挂消息当证据。

- **通用骨架 stage**:`候选 → 进行 → 等待/阻塞 → 完结`(+ `合并` / `丢弃`)。
- **字段**:`owner`、关联 persons(M:N)、关联 commitments(承诺/到期)、关联 messages(M:N 证据)、T4 诊断(沿用现有)。
- **事-类型扩展位**:通用骨架够用;特定类型若需独立 stage,走**可选扩展**,不一上来给每类各做一套(YAGNI)。**已落地两类(坐实扩展位)**:
  - **落地业务事(A)**:stage = 候选→进行→等待→**成交**。小群/1:1。
  - **营销/传播事(B)**:stage = 曝光→互动→**浮线索**→转化。大群(见 §1.5)。
  - 同一个 `lifecycle.advance()`、**两套 stage 模板**,不用两套引擎。
- **候选事**:LLM 从会话流提议新事候选,人确认后才落为正式事(HITL)。

**为什么"生命周期"是卖点(AI-native 代差)**:传统 CRM 的业务记录是被动的,靠人手动推 stage,人一忙就烂在那、被忘掉。本系统把生命周期做成**引擎**——事闲置了、承诺到期了,它主动冒出来要你处理,把"会腐烂的业务"变成"监管下自我推进的线程"。这是可营销给客户的核心价值之一。

---

## §3 生命周期引擎 `lifecycle.advance()`

把生命周期**抽象成函数**:

```
lifecycle.advance(matter, event) -> (new_stage, 副产物)
```

- **event 双源**:
  - **确定性源(无 LLM 也跑)**:承诺到期 / 闲置 N 天 / 新消息到达 / 人工。
  - **LLM 语义源(assist)**:读 timeline 提议"这事卡了 / 成了 / 该并入另一件"。
- **两源都只提议 transition,人确认(HITL)**。系统不黑箱改 stage。
- **副产物**:`撩由头候选` / `提醒` / `染色更新`。**撩的由头就是 transition 的副产物**——流水线在此闭环。
- **LLM-optional**:确定性源保证 LLM 全挂时生命周期仍推进(承诺/闲置照样冒泡),只是少了语义提议。

## §4 记忆层 `recall()` / `expand()` — 显著性检索,不是转储

**不建新库**。记忆 = 盖在现有 SQLite 真相源(persons/channels/matters/commitments/messages/media)上的**检索接口**。

```
recall(person, channel, *, now, purpose, budget) -> ContextBundle   # 默认:显著包
expand(handle, *, cursor)                         -> FullSlice       # 钻取:全量
```

**核心立场**:「Agent 像人」的真假活在这一层。人有选择地、按当下情境想起该想起的,不平铺全部记录。故记忆接口的活是**显著性(salience)函数**,不是 dump——取多了像数据库播报,取错了像冒充,都露馅。

- **`ContextBundle`(纯 SQL、零 LLM,每项带可钻取 `handle`)**:
  - `recent` — 该通道近端对话窗
  - `open_matters` — 未结的事(撩由头),handle→钻该事全程
  - `due_commitments` — 到期/逾期承诺
  - `temperature` — 染色权重 + 多久没碰(现成)
  - `persona / 话术命中线索` — 喂双闸白名单侧
- **分层**:默认返回显著包;`expand(handle)` 钻进去看该人/该事/该 thread 的**全量历史**(供人在回路 UI 钻取 + LLM 不够时深查)。
- **显著性 v1(确定性,可解释)**:`score = w·近期 + w·事未结 + w·承诺到期 + w·关系权重`,按 `budget` 截断取 top。
- **`purpose` 调权**:撩→事/承诺权重高;回复→近期权重高;意图分类→话术历史权重高。
- **时间×空间双轴可索引**:时间=近端/到期/逾期;空间=哪条通道/哪个关系账户/哪件事。
- **扩充位**:显著性是**可插拔策略**——v1 确定性;日后加 `SemanticStrategy`(embedding 语义召回)只是注册新策略,`recall/expand` 签名与下游不动。
- **LLM-optional**:`recall/expand` 纯 SQL。bundle **同时**喂确定性白名单匹配(无 LLM 也跑)与 LLM 起草/风险判;LLM 全挂 → 还能跑白名单 → 优雅降级。

## §5 监管下自动沟通流水线

一条流水线,六段:

```
染色(撩谁) → 事(由头) → 话术+记忆(怎么说) → 双闸(能不能自动) → 时间窗(何时) → 发
   §jl现成      §3副产物      §4 recall            §6              §7
```

- **被动回复**:对方先开口 → recall → 双闸 → 时间窗 → 发或交人。
- **主动撩(发起)**:染色 dispatch 出"该碰的人" + 生命周期冒出"该推进的事" → 由头 → 话术 → 双闸 → 时间窗 → 发或交人。
- **不对称原则**:**主动撩比被动回复贵**(花掉关系的"打扰额度",撩错时机负作用更大)→ 主动撩的安全区**更窄**,默认更需要人点话术。

---

## §6 双闸安全区(护栏)

「安全区内自动发、出安全区停下来交人」的判定 = **双闸**:

```
自动发 ⟺ 命中确定性白名单/意图  ∧  LLM 风险判定为低
```

- **闸一(确定性,无 LLM 也跑)**:命中"已批准话术库 / 意图分类"。
  - **★ 产品特性:安全区由用户自灌**。AMR 给一个 UI,让使用者把**自己的具体业务、要过的内容、认可的话术**灌进去 / 训练进去 → 这就是闸一的白名单料。**"安全圈是客户自己画的"** = 可营销的差异点(非通用客服腔,是客户自己的生意话)。规则/话术库**可简单起步**(先纯规则匹配),日后升级意图分类器。
- **闸二(LLM assist)**:对草稿打风险判定。
- **铁律:LLM 只能「往人那边拦」(让它更保守),永不能越过白名单去「授权发出」。** LLM 全挂 → 退回全人工 → 优雅降级,不违反 LLM-optional。
- **出安全区(新承诺/金额/情绪/未知)→ 停,交还人**(回到原 countdown 选话术模型)。

## §7 时间窗 + 主动撩:三层纵深防御(C+B,2026-06-28 定)

「能不能自动发」= 三层**与门(都过才发)**,各管不同的事,下层接住上层漏的——**不依赖检测"人在不在线"**:

| 层 | 管什么 | 机制 |
|---|---|---|
| **① 时间窗(硬外界·得体性)** | 别在不合适的点发(晚上给人发不得体,群主晚上不说话) | 窗外一律不自动发。这是社交得体,**不是** veto 机制 |
| **② 风险分层(C·无害性)= 双闸(§6)** | 窗内只自动发"看不看都无害"的(寒暄/确认/标准FAQ) | 有分量的(承诺/金额/情绪)永远等人,不管几点 |
| **③ 否决窗(B·兜底)** | 接住 ② 的分类错误 | 窗内自动发仍走 countdown 否决;边界情形推一条手机通知一键否决 |

- **悖论已解**:时间窗不负责保证"你在看"(它做不到),它负责"别在不合适的点发";"你看没看"由 ② 的无害性兜底、③ 接住误判。故**不做在场检测**(最难、最不可靠的信号被绕开)。
- **主动撩**:Agent 在窗内按记忆+话术主动撩;主动撩安全区比被动回复更窄(§5 不对称原则)。
- **授权按关系账户分档**(jl 命门):客户(错=丢钱,可补救)vs 家人(错=情感背叛,不可逆)风险不对称——**每个关系账户各自配时间窗 + 安全区档位**,不设一刀切全局开关。
- **实现难度**:① 低(每账户配时段+比较)、③ 低(countdown 已现成)、② 中(代码不难,难在"无害集合"持续调校=产品活)、在场检测=不做。

## §8 UI:数据层之上的可换皮肤(2026-06-28 王总钦定)

**最高定调:人是扳手柄的控制者,不是读屏的(王总钦定)。** 人主要在"看着"、偶尔打字调方向;UI 不只伺候人,**更伺候 Agent 的诉求**;人看得没那么多。落地三条 + 一个对称:
- **少看多控**:逐条会话是 Agent 该消化的;人看更高一层"Agent 在干嘛、哪件该我拍板"。把"要人定夺的"恰好推到眼前,其余 Agent 自跑。
- **Agent 一等公民**:UI 显式有 Agent 的**待办/草稿/lifecycle 提议队列**,每条配**一手势控制**(放行/否决/扳档/打字微调)。countdown 否决窗 = 此范式雏形。
- **打字=偶尔调方向**,非主交互;人的主动作是监管手势。
- **UI 自身遵循显著性(对称 §4)**:默认**安静控制台**(只浮该你扳的手柄,看得少),要决断时**钻进去深看全貌**——即 UI 版的 `recall(显著)/expand(全量)`。控制台 = Agent 替你 recall 出"此刻该你管的"。


**UI = 盖在稳定数据/函数层之上的可换皮肤。** 只要底层 `/api/*`(conversations / persons / matters / messages / outbox / self …)与函数稳定,上层排布可换、可并存——**不同用户按使用习惯各选各的皮肤**,不重写数据层。故:
- **先定数据契约(UI 消费什么),再做皮肤**;皮肤是纯呈现层。
- **可同时做 2–3 套皮肤**让用户挑(候选见下);新增配置面只加端点,皮肤照旧消费。
- 候选皮肤(渲染对比中,用户决定先做哪 2–3 套):
  1. **隐喻锚定矩形**:左下=我(站位/composer/全局设置)、右上=事(北极星+该事配置)、中间=人/信息·能量·物质流动场。可扫矩形,不反工学(以人为本·以终为始落地)。
  2. **真对角线**:内容沿左下→右上斜轴排,最醒目最差异化,留白多、列表难放。
  3. **双轴可切 + 隐喻点缀**:保留人视图⇄事视图,composer 固定左下、事卡高亮右上角,改动最小。
- **配置归宿沿"我→事"轴各归各位**:全局/我的设置贴左下"我";per-事/类型配置贴右上"事";per-人旋钮长在中间场的人身上——不倒进一个大设置页。

### 默认落地 = 今日简报(数字员工报告,2026-06-28 王总钦定)

控制者最常做的是"看一眼总结"。故 UI **默认落地页 = 今日简报**(不是会话列表),由**数字员工(Agent)每天提交**:
- 📈 **销售报告**(落地事 type-A 管道:进行/成交/等待 + 金额)
- 📢 **营销报告**(大群触达 + 浮出线索 + 建议拉小群,type-B)
- 🤝 **关系报告**(染色:逾期🔴/转凉🟡 + 今日该撩谁)
- 🗂 **业务进展报告**(事 lifecycle 推进 + 今日到期承诺 + 卡住的事)
- 🧭 **meta 报告**(整体健康/异常/北极星对齐)

**三级结构**:L0 今日简报(默认·看总结)→ L1 人/事双轴(从简报某项钻进)→ L2 会话/事全量(expand 深看决断)。报告是**发射台**不是终点。

**两条铁律**:
- **报告 LLM-optional**:底层 SQL 聚合拼出数字/清单(管道、逾期清单、到期承诺);LLM 只加"叙事总结"。无 LLM → 给数字清单,照看。
- **每份报告浮"需你拍板的 N 项"+ 一手势控制**(撩这4个→放行 / 拉小群→确认),否则只是读物不是控制台。

### 手机端:皮肤在桌面分叉,在手机收敛

窄屏强制竖排,桌面三套皮肤(②④③)**在手机收敛成同一竖向结构**,且恰好落地隐喻——**竖屏「我↓ 事↑」轴**:事·北极星钉顶、会话居中、我/composer 钉底(拇指区)。故手机端只设计一种,不分皮肤。三屏 + 底部 tab(简报/人/事/我):
- **简报**(家门口):`需你拍板(N)` 一拍放行/改改 + 报告卡竖排。
- **会话**(钻进去):事卡 sticky 顶部、气泡中、底部话术 chips + 倒数自动发。
- **事视图**:桌面看板的"列"→按 stage 竖向分组折叠列表。
mockup:`docs/superpowers/specs/ui-mockups/mockup-mobile.html`。

### 双轴镜头(所有皮肤共享的内容模型)

UI 镜像架构——同一份消息底料,两个镜头(人视图/事视图)。

- **人视图(像现在,LLM-free 读平面)**:左 人列表 / 中 会话 timeline / 右 该人的事(lifecycle 卡)。日常"跟谁聊了啥",人人会用。
- **事视图(lifecycle 看板,卖点)**:按 stage 列(候选/进行/等待/完结);点一件事 → 看它**横跨多人多通道的合并 timeline**。撩候选 / 到期承诺天然聚在这。
- **群/场在 UI**(§1.5):小群在事视图 = 一件多方事,在人视图 = 一个"场"条目(右栏成员按阵营分组 + 该事);**大群不进日常人视图**,单列"营销场/线索"入口——浮出线索列表 + 一键"拉小群"(建小群场+落地事+带历史)。
- **HITL 手柄**:发送走 countdown 否决窗(选 1/3 话术 → 倒计时 → 自动发,可改可撤);生命周期 transition、事-归属、候选事入库、拉小群转化都是显式确认闸。

## §9 LLM-optional 边界总表(每个组件:无 LLM 怎么降级)

| 组件 | 无 LLM 时 |
|---|---|
| `recall/expand` | 纯 SQL,照常出显著包 |
| 事-归属 | 人手动挂消息到事;少了 LLM 候选 |
| `lifecycle.advance` | 确定性源(承诺/闲置/消息/人工)照常推进;少了语义提议 |
| 双闸 | 闸一(白名单)照跑;闸二缺失 → **一律交人**(更保守) |
| 撩草稿 | 退回话术库模板填充(无生成),人选/人改 |
| UI 读/会话/事看板 | 全部可用(读平面本就 LLM-free) |

**结论:LLM 全挂 → 人完全在回路、仍能沟通,所有自动降级为手动。**

## §10 HITL 闸总表

| 闸 | 触发 | 人怎么介入 |
|---|---|---|
| 候选事入库 | LLM 提议新事 | 确认/否决/编辑 |
| 事-归属 | 消息挂到哪件事 | 确认/改挂 |
| 生命周期 transition | advance 提议改 stage | 确认/驳回 |
| 自动发(双闸通过) | countdown 否决窗 | 改话术/撤销/放行 |
| 出安全区 | 双闸不过 | 选 1/3 话术,Agent 执行 |
| 关系账户授权 | 开/关某账户自动沟通 | 分档开关 + 时间窗 |
| 拉小群转化(§1.5) | 系统提议值得转化的线索 | 确认/否决,一键建小群+落地事 |

## §11 扩充位(pluggable,先留接口不实现)

- 记忆显著性策略:`DeterministicStrategy`(v1)→ `SemanticStrategy`(embedding)。
- 事-类型生命周期模板:通用骨架 → 特定类型扩展 stage。
- 通道:契约 v1 已通道无关,新通道吐 canonical 即接入。

## §12 范围与分解(落地顺序)

这份设计拆成几份独立 implementation plan,**记忆层先行**(它是算料地基,双闸/撩/生命周期都消费它):

1. **记忆层 `recall()/expand()`**(§4)— 纯逻辑 TDD,先做。无 LLM 依赖。
2. **事生命周期引擎 `lifecycle.advance()`**(§2/§3)— 纯逻辑 TDD;确定性源先,LLM 语义源后。
3. **UI 双轴可切**(§8)— 人视图先补完整,再加事视图看板。
4. **群组/场 + 线索漏斗**(§1.5)— 场实体 + 阵营标签 + 营销事-type B + 线索转化(系统提议+人确认)。依赖 1/2,UI 部分依赖 3。
5. **双闸 + 自动沟通流水线**(§5/§6/§7)— 最后,且**主动撩默认关、逐账户开**。
6. **远程安全接入**(§14)— EasyTier mesh(部署)+ AMR web TLS(stdlib `ssl`)。独立 phase,与 UI 互不阻塞。

每份各自 spec→plan→实现。**先讨论再实现**:本设计为正本,具体接口细节在各 plan 里钉死。

## §13 不做(YAGNI)

- 不做"每类事一套生命周期"(通用骨架 + 按需扩展;v1 只两类 A/B)。

## §14 生产部署·远程安全接入(2026-06-28 王总钦定,作为开发任务)

用户带手机离开 AMR 部署机的 LAN 后,要能**安全回连**。分两层,各司其职:

- **IP/链路层 — EasyTier mesh(部署配置,非 AMR 应用代码)**:AMR 机 + 手机加入 EasyTier 虚拟网,手机经虚拟 IP 回连,**无需公网暴露 / 端口转发**。AMR 应用不感知 VPN——只管绑接口,可达性交给 EasyTier。→ 落运维手册。
- **应用层 — TLS(AMR 代码改动,stdlib 零新依赖)**:AMR web(`http.server`)加 TLS(stdlib `ssl.SSLContext`):
  - 默认 **自签证书**(个人/小队;浏览器首次告警可接受),附生成脚本;
  - 可选 **FQDN + 真证书**(接 §6/settings 里已有的 FQDN 后端配置)。
- **质疑/口径**:EasyTier 已加密链路,**TLS 是应用层纵深防御** + 解锁浏览器 HTTPS-only 能力(剪贴板/PWA/通知);不是多余。token 认证保留(走 TLS 后不再明文上线);`token-in-URL` 仍入历史/referer = 小风险,后续可挪 header/cookie(非阻塞)。
- **落地**:作为 implementation plan 一个**独立 phase**(UI 之后可做、互不阻塞);运维手册同步写 EasyTier 接入 + 证书轮换。

## §15 认证·授权·订阅(计费)(2026-06-28 王总钦定,后续阶段,框架待沟通)

访问 AMR 界面前端加一层:
- **认证**:登录界面;**注册用 Email 即可**(email + 验证)。
- **授权**:登录后的访问控制(谁能看/操作哪些资源)。
- **订阅/计费**:把订阅做起来(套餐/付费墙)。**后台订阅设计框架另约时间与王总沟通**——本节先占位,不在此细化。
- **架构跃迁提醒(质疑)**:这把 AMR 从「单用户 + 共享 token(本地/LAN)」推向「多用户 + 账户 + 付费」SaaS——真实的架构跨度(多租户隔离、每用户数据边界、支付合规)。与 §14 叠加:**远程可达 + 多用户 ⇒ 共享 token 不够,需真正的每用户认证**。故顺序上认证/授权应先于或并于远程开放。
- **安全红线(实现时遵守)**:支付凭据/卡号等由用户在**专门支付渠道(第三方支付服务商)**输入,AMR **不经手明文卡号/不自建收银台**(PCI 思路);AMR 只对接支付服务商的会话/回调。
- **落地**:后续独立 plan(认证 → 授权 → 订阅),**订阅框架待专门沟通后再设计**。
